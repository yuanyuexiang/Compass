## 1. 设计目标与原则

| 原则 | 说明 |
| --- | --- |
| **公共层与租户层分离** | 采集、清洗、附件解析、AI 结构化是**租户无关**的公共流水线，每条公告只处理一次；能力匹配、评分、推荐、通知是**租户相关**层，按租户各算各的。这是控制 LLM 成本和支撑多租户的核心设计。 |
| **模块化单体起步** | V1 用单仓库、单应用 + 异步 Worker 的模块化单体，按模块划分代码边界（对应说明书 §8 的十个模块），不上微服务。模块间通过接口和消息队列解耦，未来可按模块拆分服务。 |
| **模型可替换** | LLM 调用统一经 LiteLLM（不自研封装层），DeepSeek 为默认模型，通过 Router 配置可切换/混用其他模型（说明书"可扩展"要求）。 |
| **AI 输出可追溯** | 所有 LLM 提取和风险判断必须附带原文引用与置信度，防幻觉，可人工复核（对应说明书 §5.2 质量保障）。 |
| **流水线可重放** | 每个阶段的输入输出落库，任一阶段失败可单独重试，模型/Prompt 升级后可对历史数据重跑。 |

**V1 非目标**：标书（招标文件）深度解析与投标方案生成（V2）、知识图谱与竞对分析（V3）。

## 3. 技术选型

| 层 | 选型 | 理由 |
| --- | --- | --- |
| 后端语言/框架 | **Python 3.12 + FastAPI** | 采集、文档解析、OCR、LLM、向量生态全在 Python；FastAPI 异步适合 IO 密集的 LLM/爬虫场景 |
| 异步任务 | **Celery + Redis**（Beat 做定时） | 流水线各阶段即 Celery 任务链；成熟、可观测、支持重试与优先级队列 |
| 数据库 | **PostgreSQL 16 + pgvector** | 一库同时承担关系数据 + 向量检索 + JSONB 半结构化存储，V1 避免引入独立向量库；数据量大后可平滑迁 Milvus/Qdrant |
| 缓存/队列 | **Redis 7** | Celery broker、去重指纹缓存、限流 |
| 采集 | **httpx 优先 + Playwright 兜底** | 静态页/JSON 接口用 httpx，轻量高并发；仅 JS 渲染、JS 校验类站点用 Playwright（对比 Selenium：原生 async、自动等待、多 context 并发更省资源；不引入 Scrapy：调度/去重/管道本架构已自建） |
| PDF 解析 | **PyMuPDF** | 快、对中文友好，可判断是否扫描件 |
| Word 解析 | **python-docx**（.doc 老格式经 LibreOffice 转换） | 覆盖政采附件主流格式 |
| OCR | **PaddleOCR 3.x（PP-OCRv6 中文模型）** | 开源免费、中文识别效果好、可本地部署；仅对扫描件兜底触发。安装为两步：paddlepaddle（官方源）+ `paddleocr[all]` |
| LLM | **DeepSeek deepseek-v4-flash 为主（批量提取/精排）；复杂风险分析开思考模式或换 deepseek-v4-pro** | 已具备接入；成本极低适合大批量公告处理；OpenAI 兼容协议；1M 上下文。⚠️ 旧模型名 `deepseek-chat`/`deepseek-reasoner` 于 2026-07-24 弃用，代码一律使用 v4 模型名 |
| LLM 网关 | **LiteLLM（直接使用，已确认不自研封装）** | 统一 OpenAI 兼容接口；Router 做多模型路由与重试；success_callback 做按租户/环节用量统计，满足"可更换 AI 模型" |
| Embedding | **bge-m3**（V1 经 SiliconFlow 等 API 调用，量大后自部署） | DeepSeek 不提供 embedding API；bge-m3 中文效果好、支持长文本，是国内主流选择 |
| 对象存储 | **MinIO**（S3 协议，已确认） | 附件与原文存储；后续上云可平滑换 OSS/COS |
| 前端 | **Next.js（React + TypeScript）+ Ant Design** | 用户指定（生态最主流）；管理后台以 CSR 为主，SSR 非必需；Docker 部署用 standalone 输出 |
| 部署 | **Docker Compose**（单机 4C16G 起步），预留 K8s | V1 单机即可跑通全链路 |

## 4. 核心流水线设计（公共层）

流水线 = Celery 任务链，每阶段幂等、可单独重试，状态机落在公告记录上：
`crawled → cleaned → attachments_parsed → ai_extracted → embedded → published`。

### 4.1 采集模块

**数据源策略（已确认 2026-07-04）：官方枢纽自采为主，商业 API 补盲为辅，不爬商业聚合站。**

- **首批接入（M1）官方枢纽** —— 枢纽站本身即聚合节点，少量站点覆盖大盘：
  1. 中国政府采购网 `ccgp.gov.cn`（政府采购法定发布渠道，已实测纯 httpx 可采）；
  2. 全国公共资源交易平台 `deal.ggzy.gov.cn`（汇聚各省市公共资源交易分平台）；
  3. 中国招标投标公共服务平台 `bulletin.cebpubservice.com`（国家级互联互通平台，免登录可检索，以工程招投标为主）；
  4. **江苏省**（种子客户所在地，已确认）：江苏政府采购网 `zfcg.czt.jiangsu.gov.cn` 及省公共资源交易平台（生产网络下可采集性待复测，见附录 D）。
- **长尾补盲**：上万个代理机构/央企平台官网不自建适配器，按需采购商业数据 API（剑鱼标讯、采招网等需商务议价；企查查开放平台/阿里云云市场按调用计费，可低成本试水），以统一的 `CommercialApiAdapter` 接入同一条流水线。
- **红线**：不采集标800、千里马、剑鱼等商业聚合网站 —— 直接竞争关系 + 绕过反爬 + 实质性替代三要件俱全，属不正当竞争高风险（可比判例判赔 50万–300万元）。
- 所有公告在产品内**注明原始出处链接**（合规要求 + 用户信任）。

架构设计：

- **源适配器（SourceAdapter）插件化**：每个平台实现统一接口 `list_announcements(since) / fetch_detail(url)`，返回统一的 `RawAnnouncement`。新增平台 = 新增一个 Adapter 类 + 一条源配置记录，不动框架代码。
- **httpx 优先、Playwright 兜底**的适配策略：
  1. 站点为静态页或有可直连的 JSON 接口 → 纯 httpx Adapter；
  2. 动态渲染站点 → 先用 Playwright 的**网络拦截**探路，抓到页面自身调用的 XHR/JSON 接口后，降级为 httpx 直连接口；
  3. 确实无法绕开浏览器的（JS 设 Cookie、前端加密参数）→ Playwright Adapter，控制并发数。
- 源配置表管理：抓取频率（cron）、并发数、请求间隔、UA/代理配置、启用状态。
- **去重**：URL 归一化指纹 + 标题相似度（SimHash）双重去重，Redis 短期指纹 + DB 唯一约束兜底。
- **反爬与合规**：全局限速（每源可配 QPS）、随机间隔、失败退避；只采集公开公告页面；代理池作为可选组件后接。
- **失败告警**：连续 N 次采集失败自动通知管理员（源站改版的信号）。

### 4.2 清洗与附件解析

- 清洗：HTML 正文提取（去导航/页脚）、编码归一、字段初步正则抽取（作为 LLM 提取的交叉校验）。
- 附件：下载存对象存储，按类型分发解析器；PDF 文本层字符数低于阈值判定为扫描件 → 走 PaddleOCR。
- 解析产物统一为**带页码/位置信息的纯文本**，供 LLM 使用并支撑"原文引用"。

### 4.3 AI 理解（字段提取 + 智能分类）

- 一次 LLM 调用完成说明书 §5.2 的 12 个字段提取 + §5.3 的两级分类，**JSON 结构化输出**（DeepSeek JSON 模式，官方提示偶发返回空 content），Pydantic 严格校验，空返回/不合法自动重试（最多 2 次，仍失败进人工队列）。
- 每个字段返回 `{value, evidence(原文片段), confidence}`；低置信度字段在后台标黄供人工复核，修正数据回流为 Prompt 优化的评测集。
- 正文 + 附件文本超长时：先分段摘要再提取（map-reduce）；预算、截止时间等关键字段用正则预抽取结果与 LLM 结果交叉验证，冲突则降置信度。
- 提取完成后生成**项目摘要文本**（标题 + 采购内容 + 行业 + 地区 + 预算的规范化描述），用 bge-m3 生成向量入 pgvector。

## 5. 租户层设计

### 5.1 企业能力画像（企业知识库）

画像 = 结构化字段 + 非结构化语料两部分：

- **结构化**：产品线、服务类型、覆盖行业、覆盖区域、资质证书（如 ISO/CMMI/建筑智能化资质）、合作品牌、注册资本等 —— 用于硬性过滤和风险判断。
- **非结构化**：公司简介、成功案例描述、历史投标项目 —— 切块后 embedding 入向量库（按 tenant_id 隔离），用于语义匹配。
- 画像支持后台表单维护 + 上传公司介绍文档由 LLM 自动抽取初始化（降低冷启动成本，对应说明书 §5.4）。

### 5.2 匹配引擎（三级漏斗：规则 → 向量 → LLM）

每条新公告 `published` 后，对每个订阅租户触发匹配任务：

1. **规则硬过滤（零成本）**：区域黑白名单、预算区间、明显不相关大类 → 直接淘汰，预计过滤 70%+。
2. **向量粗排（低成本）**：公告向量 × 该租户画像/案例向量，取相似度 top 分位（阈值可调）进入精排，控制精排量在每租户每天几十条量级。
3. **LLM 精排（DeepSeek）**：Prompt 输入 = 项目结构化信息 + 租户画像摘要，输出评分卡 JSON：
   - `match_score`（0–100）与 ★ 推荐等级（映射规则可配，如 ≥90 → 五星）；
   - `match_reasons[]`：哪些能力对上了，引用画像与公告原文；
   - **风险分析（说明书 §5.6 六项）**：品牌限制 / 排他条件 / 特殊资质缺失 / 预算不足 / 竞争激烈 / 废标风险，每项 `{命中与否, 依据原文, 严重程度}`；
   - `advice`：参与建议（建议参与 / 谨慎参与 / 不建议参与）。
4. 结果落 `match_results` 表，驱动推荐列表与通知。

> 复杂大项目（预算高于阈值或附件多）可开启思考模式（或路由到 deepseek-v4-pro）做深度风险分析，量小成本可控。注意：**思考模式不支持 Function Calling**，结构化输出统一走 JSON Output。精排 Prompt 将固定的租户画像摘要置于前缀，利用 DeepSeek 上下文硬盘缓存自动命中降本（命中价约为未命中价的 1/50）。

### 5.3 AI 自然语言搜索

"查找江苏省预算超过300万的 AI 项目" 的实现：

1. LLM 将口语查询解析为**结构化查询 DSL**：`{region:"江苏", budget_min:3000000, category:"AI", ...}` → 转 SQL 条件过滤；
2. 剩余语义部分（如"AI 项目"这类模糊概念）走向量检索补充召回；
3. 合并去重后按匹配度排序返回。DSL 解析失败时降级为纯语义搜索，保证有结果。

## 6. 数据模型（核心表）

```
公共层：
  sources            采集源配置（平台、cron、限速、适配器名）
  announcements      原始公告（url、原文HTML、状态机、去重指纹）
  attachments        附件（文件、解析文本、是否OCR）
  projects           结构化项目（12字段+分类+evidence JSONB、摘要、向量）

租户层（均含 tenant_id）：
  tenants / users            租户与账号（RBAC：平台管理员/租户管理员/销售）
  company_profiles           企业能力画像（结构化字段 JSONB）
  profile_chunks             画像语料切块 + 向量
  match_results              匹配结果（score、星级、理由、风险JSON、建议）
  subscriptions              推送规则（星级阈值、行业/区域过滤、渠道、静默时段）
  notifications              通知记录（渠道、状态、重试次数）
  follow_ups                 销售跟进状态（V1 简化为：待看/跟进中/放弃/已投标）
```

**多租户隔离**：共享库共享 Schema + `tenant_id` 行级隔离，API 层从 JWT 解析租户上下文强制注入查询条件，Postgres RLS 作为第二道防线。公告/项目数据全局共享（不含租户信息），匹配结果起才带 tenant_id —— 兼顾成本与隔离。

## 7. 通知中心

- 统一 `Notifier` 接口 + 渠道适配器：**邮件(SMTP)、企业微信(群机器人 webhook)、钉钉(webhook)、飞书(webhook)、Web 站内信（SSE 实时推送）** —— 以上 V1 全部落地，接入成本低。
- **个人微信**：无官方推送 API，需公众号服务号模板消息（需企业资质与审核），**已确认 V1 暂缓**、资质就绪后接入（§13 决策点 4）。
- 两种触发：
  - **每日商机日报**（默认每早 8:30）：昨日采集总数、AI 推荐列表按星级分组，一条消息汇总；
  - **重大项目即时提醒**：满足订阅规则（如 ≥四星 或 预算≥N）匹配完成后立即单条推送。
- 发送失败指数退避重试 3 次，全渠道失败落站内信兜底。

## 8. Web 管理后台（Next.js + Ant Design）

V1 页面清单：

1. **商机工作台**：今日推荐（星级分组卡片）、匹配理由与风险标签展开、跟进状态流转；
2. **商机查询**：多维筛选 + AI 自然语言搜索框，项目详情页展示结构化字段、原文对照、附件预览；
3. **企业画像管理**：画像表单 + 文档导入抽取 + 案例库维护；
4. **订阅与通知设置**：推送规则、渠道绑定、静默时段；
5. **采集运营（管理员）**：源配置、采集状态监控、失败告警、低置信度字段人工复核队列；
6. **租户与用户管理（平台管理员）**。

## 10. 非功能设计

### 10.1 成本控制（LLM 是最大变动成本）

- 公共层提取每公告只算一次，全租户共享 —— 最大的省钱设计；
- 匹配用"规则过滤 → 向量粗排 → LLM 精排"漏斗，LLM 只看漏斗尖；
- 精排 Prompt 固定前缀（租户画像摘要）利用 DeepSeek 上下文缓存自动降本；
- 以说明书示例日 862 条公告、每条正文+附件平均 6K tokens 估算，提取环节日均约 600 万 tokens 输入。按 **deepseek-v4-flash 现价（2026-07 官网核实：输入未命中 ¥1/百万、缓存命中 ¥0.02/百万、输出 ¥2/百万）**，公共层提取日成本约 **¥6–10 人民币**；每新增一个租户仅增加精排增量（日几十条 × 2K tokens，且画像前缀高缓存命中，可忽略不计）。Embedding 按 ≤¥1/百万 tokens 估算（SiliconFlow bge-m3 有免费档，DashScope/智谱约 ¥0.5/百万）。
- LLM 网关统一记录 token 用量，按租户/环节出成本报表。

### 10.2 质量与可观测

- 建立**评测集**：人工标注 100–200 条公告的标准答案，Prompt/模型变更前后跑评测（字段准确率、分类准确率、风险判断查全率），防止改 Prompt 退化；
- 结构化输出全部 Pydantic 严格校验；关键字段（预算、截止时间）正则交叉验证；
- 日志/指标：各阶段处理量、失败率、时延、token 消耗，V1 用 Prometheus + Grafana（Docker Compose 内置）。

### 10.3 安全

- **认证与授权**：JWT 认证；RBAC 三角色（平台管理员 / 租户管理员 / 销售）；租户上下文由中间件强制注入，Postgres RLS 兜底；
- **数据访问**：附件下载走带过期时间的签名 URL，不暴露存储路径；API 按用户/租户限流；
- **密钥管理**：LLM API Key、各通知渠道 webhook 密钥走环境变量/密钥管理，不入库明文；
- **审计**：登录、画像修改、订阅变更、人工复核操作记审计日志。

### 10.4 合规提示

- 仅采集**政府/官方公开发布**的公告页面，遵守目标站 robots 与访问频率约束；采集行为限速、可随时熔断，不对目标系统造成过载；
- **不采集商业聚合平台**（标800、千里马、剑鱼等）的数据（红线，见 §4.1；数据抓取不正当竞争判例判赔 50万–300万元量级）；
- 附件与原文注明来源与原始链接，平台定位为线索发现而非内容转载。
